MONI - Product and Data Design Brief

1) Product context
- MONI is an uptime-focused monitoring product built as a single-team project.
- Core user value: add a target -> monitor availability/performance -> receive alerts on failure/recovery.
- Current production model is intentionally simple: one tenant scope per account, no role/package hierarchy.

2) Solution architecture (current baseline)
- Frontend: Vue SPA.
- Backend API: FastAPI (REST, layered architecture).
- Database: PostgreSQL (single primary database).
- Async runtime: Redis + Celery worker + Celery Beat scheduler.
- Deployment: Docker on VPS with reverse proxy and TLS.
- Messaging note: RabbitMQ is intentionally deferred; Redis is the only queue backend in this phase.

3) Functional scope
- Authentication:
  - Email/password registration and login.
  - Email verification via Gmail SMTP.
  - Google OAuth login.
  - SPA session with short-lived access token + refresh token flow.
- Monitoring:
  - Target types in schema/API: HTTP(S), TCP, ICMP.
  - Delivery order: HTTP first, then TCP, then ICMP (ICMP needs OS/container capability handling).
- Alerting:
  - Email-only channel in MVP.
  - Additional channels are planned after core monitoring quality is stable.
- Product boundaries:
  - No public status page in this phase.
  - No SEO/public growth scope in this phase.

4) Business analysis framing
- Primary actor: authenticated end user who owns monitors.
- Primary jobs-to-be-done:
  - Create and maintain monitors.
  - Detect outages/performance degradation quickly.
  - Investigate checks/incidents history.
  - Receive actionable failure/recovery alerts.
- Success criteria:
  - Reliable state transitions (up/down/slow/checking/pending/paused).
  - Deterministic run-check behavior and scheduler behavior.
  - Inspectable history for operational decisions.

5) Database structure overview (conceptual)
- `users`: account identity and authentication ownership.
- `refresh_tokens`: token/session continuity and revocation support.
- `monitors`: target configuration and current snapshot status.
- `check_runs`: immutable execution records for each monitoring attempt.
- `incidents`: outage lifecycle (opened/closed) derived from check transitions.
- `alert_events`: notification audit trail (channel, event type, send status).

Key relationships:
- One user -> many monitors.
- One monitor -> many check_runs.
- One monitor -> many incidents.
- One incident -> many alert_events.
- One monitor -> many alert_events (directly queryable for operations history).

Design intent:
- Keep snapshot fields on `monitors` for fast list/dashboard rendering.
- Keep event/history tables (`check_runs`, `incidents`, `alert_events`) append-oriented for auditability and analytics.
- Use soft-delete behavior where needed to preserve historical context.
